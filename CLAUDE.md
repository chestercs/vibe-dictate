  VibeVoice ASR (Microsoft's open-source ~7B speech model).

  VibeVoice-ASR supports.

## How to operate

### Starting a build

```bash
# from the parent VibeVoice repo root (one level up):
taskkill //IM vibe-dictate.exe //F 2>/dev/null; true   # release exe lock
docker compose -f docker-compose-vibedictate-build.yml run --rm vibedictate-build
```

Output: `tools/vibe-dictate/target/x86_64-pc-windows-msvc/release/vibe-dictate.exe`.

The `taskkill` is necessary if the user has the previous build running —
otherwise the linker hits "Input/output error" trying to overwrite the
locked exe.

## Architecture notes

### Event flow

1. tao event loop ticks every ~33ms (`AppEvent::Tick`).
2. Each tick drains:
   - `GlobalHotKeyEvent::receiver()` for keyboard hotkey events.
   - `mouse_hook::MouseHookHandle::rx` for mouse-button hotkey events.
   - `MenuEvent::receiver()` for tray menu clicks.
3. Both kb and mouse events fold into a single `enum PushAction
   { Press, Release }` queue.
4. Press → start audio capture (cpal Stream lives on a worker thread).
5. Release → stop capture, encode WAV, hand to a network worker thread.
6. Network worker → `gradio::transcribe()` → text → `injector::*`.

### BindingManager

Single owner of the active hotkey routing. `apply(binding_str)` first
calls `disable()` (tear down kb + clear mouse), then routes:
- If `mouse_hook::parse_mouse_binding(s)` returns `Some` → mouse.
- Else → keyboard via `parse_hotkey` + `GlobalHotKeyManager::register`.

Exactly one of (kb_current Some, mouse_shared Some) is ever set. Keep
this invariant — it's why dual-routing bugs don't happen.

### Mouse hook is always-on

The `WH_MOUSE_LL` hook installs at startup and stays installed. It
checks `mouse_shared` on every qualifying event and fires nothing if
the binding is keyboard-only. Per-event cost is negligible. Don't try
to install/uninstall the hook based on the active binding — race
conditions during rebind would drop events.

The hook **does not suppress** mouse events. Middle-click still pastes,
back/forward side buttons still navigate. Push-to-talk is additive.

### Why the capture popup is a separate Win32 thread

`tray-icon` and `tao` don't deliver raw key-down events to a window we
didn't register through them. So `hotkey_capture.rs` spawns a worker,
registers its own window class, runs its own `PeekMessageW` pump, and
sends the captured binding back via mpsc. Don't try to fold this into
the main event loop.

### `cpal::Stream` is `!Send`

The recording worker must encode the WAV locally and hand the bytes
(`Vec<u8>`, which *is* Send) to the network worker. Don't pass the
stream across threads — the type system enforces this.

### TLS / CA handling

`gradio::GradioClient::new` reads `extra_ca_cert` from the config and
adds it to reqwest's rustls store on top of system roots. Single PEM
or concatenated bundle — both work.

### Why blocking reqwest, not async

The app is single-threaded around tao with mpsc-coupled workers.
Blocking IO inside a worker thread is simpler than pulling in tokio
for one HTTP call per dictation. Don't refactor to async unless
there's a concrete need.

## Conventions

- **Comments**: only when WHY is non-obvious. No "// added for X" or
  "// fixes #123" — that belongs in commit messages and rots in code.
  Multi-paragraph docstrings only on public modules / structs where
  it's meaningfully helpful.
- **Logging**: `log::info!` for state transitions and successful
  operations, `log::warn!` for partial failures and degraded paths,
  `log::error!` for hard failures. Default filter is `info`.
- **Error handling**: `anyhow::Result` everywhere, `.context()` on every
  fallible IO/Win32 call so the log message tells you *what* failed.
- **No backwards-compat shims**: this is a v0.1 personal tool, not a
  shipped library. If a config field gets renamed, just rename it and
  add a one-shot migration in `Config::migrate_in_place` if needed.
- **No automated tests**: don't add a test framework "just in case".
  When the user asks for a fix, ship it; smoke-testing happens in real
  Windows apps.
- **Keep the exe small**: avoid heavy crates (no UI frameworks, no
  tokio, no serde_yaml, etc.). Two Win32 modals are cheaper than a
  GUI dependency.

## Common gotchas

| Symptom | Cause / fix |
|---|---|
| `error: linking with cl.exe failed: Input/output error` during build | Old `vibe-dictate.exe` still running; kill it first. |
| SendInput silently does nothing | Target window is admin-elevated (UIPI) or filters injected input. Logged as `SendInput partial: …` in v0.1+. |
| Hotkey "doesn't fire" | Some other app grabbed it (Discord push-to-talk, OBS, GeForce Experience). Pick a different binding. |
| Alt-based hotkey acts weird on Hungarian/German layouts | AltGr = Ctrl+Alt; Alt-only bindings get auto-migrated to F8. Use `Ctrl+Alt+...` if you really need Alt. |
| Tray icon stuck on red after cancel-flash | `flash_until` not cleared on next press. Already fixed; if it recurs, it's a regression in `main.rs` flash handling. |
| Hungarian transcription comes back in English | `language_hint` not set, or `context_info` doesn't anchor language. Defaults handle this — check config wasn't reset. |
| `cpal` panics on device init | The configured `audio.mic_device` no longer exists (mic unplugged). Reset to system default. |

## Things NOT to do

- **Don't push to `origin` of the parent VibeVoice repo** — that's
  Microsoft's. The vibe-dictate folder is mirrored separately.
- **Don't add tokio**. The blocking architecture is intentional.
- **Don't add a UI framework**. Two Win32 modals are the budget.
- **Don't bundle the VibeVoice model**. Backend hosting is the user's
  problem, vibe-dictate is a thin client.
- **Don't bypass UIPI**. SendInput-into-admin-windows requires running
  vibe-dictate as admin too. Document it; don't try to work around it.
- **Don't add `.unwrap()` in error paths**. Use `?` + `.context()`.
- **Don't add comments that describe WHAT the code does**. The code
  describes what. Comments are for why.

## Recent decisions worth remembering

- **SendInput is batched**, not per-character. Per-character hammering
  raced with browser/terminal message pumps and dropped events.
  See `injector::send_input_text` — single call, chunked at 100 events,
  return-value checked.
- **Append Enter is a separate toggle**, independent of output mode. Both
  Clipboard and SendInput modes call `injector::send_enter` afterwards
  if `output.send_enter` is true.
- **Mouse hook is process-wide** via `OnceLock<HookState>`. Windows
  doesn't give a user-data pointer to low-level hooks, so this is the
  standard workaround. Don't refactor to a per-handle pattern.
- **The capture popup window class is re-registered on each invocation**
  (returns 0 with `ERROR_CLASS_ALREADY_EXISTS` on subsequent calls,
  which is benign). No `OnceLock` needed — the cost is negligible and
  the module stays self-contained.
